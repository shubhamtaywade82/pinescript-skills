# Alerts

TradingView alerts run 24x7 on cloud servers and do not require users to be logged in to execute. 

While some alerts (such as generic price cross alerts or strategy order fills) can be created directly from the chart user interface (UI) without custom coding, programmers can implement specific alert triggers in their Pine Script® code to create highly customized alert events.

> [!IMPORTANT]
> Pine Script code cannot directly spawn a running alert on the TradingView servers. It merely defines **alert events** which script users must activate from the chart's "Create Alert" dialog panel.

---

## Alert Triggers Comparison

Pine Script® offers three main programmatic methods to define alert events:

### 1. The `alert()` Function
*   **Best Practice**: Works in both indicators and strategies.
*   Supports **dynamic strings** computed at runtime (e.g. including current prices in the message).
*   Can be placed inside conditional blocks (`if`/`switch`) in local scopes.

### 2. Strategy Order Fill Alerts
*   Triggers when the broker emulator fills an order generated by `strategy.*()` calls.
*   Programmers can specify custom payload strings for execution using the `alert_message` parameter inside order placement calls.

### 3. The `alertcondition()` Function
*   Works in **indicators only** (ignored inside strategies).
*   Must be placed in the global scope (column zero).
*   Requires static `const string` messages (dynamic variables can only be included via preset placeholders).

## Script Alerts via `alert()`

The signature of `alert()` is:
```pinescript
alert(message, freq)
```

### Parameters:
*   **`message`** (`series string`): The dynamic text payload.
*   **`freq`** (`input string`): The execution frequency limit. Options:
    *   `alert.freq_once_per_bar` (Default): Triggers only on the first call in the realtime candle.
    *   `alert.freq_once_per_bar_close`: Triggers only on the closing iteration of the realtime bar if the condition remains true.
    *   `alert.freq_all`: Triggers on every execution tick.

> [!NOTE]
> Since strategies recalculate only on bar closes by default, any `alert.freq_all` or `alert.freq_once_per_bar` calls inside a strategy will execute at most once per bar close unless `calc_on_every_tick = true` is declared in the strategy header.

### 1. Centerline Cross Alerts Example
This indicator triggers distinct dynamic alerts when RSI crosses the centerline:
```pinescript
//@version=6
indicator("All `alert()` calls")

r = ta.rsi(close, 20)
xUp = ta.crossover(r, 50)
xDn = ta.crossunder(r, 50)

if xUp
    alert("Go long (RSI is " + str.tostring(r, "#.00)"))
else if xDn
    alert("Go short (RSI is " + str.tostring(r, "#.00)"))

plotchar(xUp, "Go Long",  "▲", location.bottom, color.lime, size = size.tiny)
plotchar(xDn, "Go Short", "▼", location.top,    color.red,  size = size.tiny)
hline(50, "Centerline", color = color.gray)
plot(r, "RSI")
```

### 2. Selective Alerts (User Options)
To allow users to create separate alerts (e.g. only longs or only shorts) from a single script, filter the triggers using boolean input settings:
```pinescript
//@version=6
indicator("Selective `alert()` calls")

detectLongsInput  = input.bool(true,  "Detect Longs")
detectShortsInput = input.bool(true,  "Detect Shorts")
repaintInput      = input.bool(false, "Allow Repainting")

r = ta.rsi(close, 20)
xUp = ta.crossover(r, 50)
xDn = ta.crossunder(r, 50)

// Wait for bar confirmation if repainting is disabled
enterLong  = detectLongsInput  and xUp and (repaintInput or barstate.isconfirmed)
enterShort = detectShortsInput and xDn and (repaintInput or barstate.isconfirmed)

if enterLong
    alert("Go long (RSI is " + str.tostring(r, "#.00)"))
else if enterShort
    alert("Go short (RSI is " + str.tostring(r, "#.00)"))

plotchar(enterLong,  "Go Long",  "▲", location.bottom, color.lime, size = size.tiny)
plotchar(enterShort, "Go Short", "▼", location.top,    color.red,  size = size.tiny)
hline(50)
plot(r)
```

## Strategy Order Fill Alerts

Strategies trigger order fill alerts when simulated trades execute in the broker emulator. You can define custom strings for each order type using the `alert_message` parameter inside `strategy.entry()`, `strategy.order()`, `strategy.exit()`, or `strategy.close()` calls:

```pinescript
//@version=6
strategy("Strategy using `alert_message`")

r = ta.rsi(close, 20)
xUp = ta.crossover(r, 50)
xDn = ta.crossunder(r, 50)

// Set custom execution messages for the broker emulator
if xUp
    strategy.entry("Long", strategy.long, stop = high, alert_message = "Stop-buy executed (stop was " + str.tostring(high) + ")")
else if xDn
    strategy.entry("Short", strategy.short, stop = low, alert_message = "Stop-sell executed (stop was " + str.tostring(low) + ")")

plotchar(xUp, "Go Long",  "▲", location.bottom, color.lime, size = size.tiny)
plotchar(xDn, "Go Short", "▼", location.top,    color.red,  size = size.tiny)
hline(50)
plot(r)
```

> [!IMPORTANT]
> When using `alert_message` inside order placement code, script users must specify the `{{strategy.order.alert_message}}` placeholder in the "Message" field of the TradingView "Create Alert" dialog. Otherwise, custom strings will not map to the final alert payload.

## `alertcondition()`

The `alertcondition()` function registers distinct alert events selectable in the "Condition" field of the chart alert panel.

The signature is:
```pinescript
alertcondition(condition, title, message)
```

### Rules & Restrictions:
*   **Scope**: Must be called in the global scope (column zero).
*   **Static Message**: The `message` parameter must be a constant string (`const string`) known at compile time. Dynamic data can only be included via placeholders.

### Placeholders

The following placeholders are evaluated at runtime inside `alertcondition()` messages:

*   `{{exchange}}`: Exchange name (e.g. `NASDAQ`, `BINANCE`).
*   `{{ticker}}`: Symbol name (e.g. `AAPL`, `BTCUSD`).
*   `{{interval}}`: Chart timeframe (e.g. `D`, `60`).
*   `{{open}}`, `{{high}}`, `{{low}}`, `{{close}}`, `{{volume}}`: Bar values at trigger execution.
*   `{{time}}`: Timestamp at bar start (UTC ISO 8601).
*   `{{timenow}}`: System clock timestamp when the alert triggers.
*   `{{plot_0}}` to `{{plot_19}}`: Value of the corresponding plot index (numbered 0-19 in order of appearance).
*   `{{plot("[plot_title]")}}`: References a specific plot by its title:
    ```pinescript
    // Double quotes inside single-quoted message string
    alertcondition(xUp, "Long Alert", message = 'RSI crossed 50 at: {{plot("RSI")}}')
    ```

## Avoiding Alert Repainting

Alerts only execute on realtime bar updates. If a trigger is evaluated on realtime ticks before a bar closes, it may fire an alert on a condition that eventually disappears once the bar confirms.

### To Prevent Repainting Alerts:
1.  Configure the alert in the charts UI to execute **Once Per Bar Close**.
2.  Alternatively, check `barstate.isconfirmed` in your conditional code logic before executing the `alert()` call:
    ```pinescript
    if crossCondition and barstate.isconfirmed
        alert("Confirmed Signal!")
    ```
